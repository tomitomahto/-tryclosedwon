The Real Story

I built tryclosedwon.com in one session while actively job searching. What you're reading is pulled directly from that conversation, not summarised or polished after the fact.

The prompts below are organised by what they were actually trying to accomplish — not by topic or category. Some of them will look simple. That's the point. The most effective prompts usually are.

The dark version existed before this session. The white pivot happened in the first message. The real lesson is knowing what you want before you start typing.
Part 1 — Starting with clarity, not vibes
The first mistake most people make when building with AI is starting with "make it look cool." These prompts show what happens when you start with a specific reference point instead.
1
The design pivot in one sentence Real Exchange
This was the actual opening message that set the entire visual direction for the product.
Me
The design feels too edgy and dark. I want it to match sites like LinkedIn and Built In. Clean, white, professional. Familiar enough that people know exactly how to use it without thinking.
Claude
Got it. Clean, white, professional. Familiar enough that users trust it immediately, distinctive enough to be memorable. Let me rebuild it.
Why this worked
Naming specific reference sites ("LinkedIn and Built In") gave the model something to calibrate against. "Too edgy and dark" alone leaves room for interpretation. Adding the reference removed the guesswork. Short, specific, and directional beats long and descriptive every time.
2
Getting clean cards without saying "clean" Prompt Pattern
When the initial cards felt cluttered, this is the exact prompt that fixed them — without asking for a redesign.
Me
The cards feel cluttered. Too many tags, too many actions. What's the minimum information someone needs to decide whether to click into a role?
Claude
Good instinct on the money. Showing OTE on every card makes it look like a salary aggregator rather than a premium career platform. Less is more. The minimum: title, company, location, work type, seniority level, and a salary bracket (not the full number). Actions: AI button and Apply. Everything else lives in the detail panel.
The pattern here
Instead of "make it less cluttered", ask what the minimum viable information is. This forces the model to think about hierarchy rather than just removing things at random. The answer usually reveals a design principle you can apply everywhere.
3
The salary display debate Real Exchange
This exchange shaped one of the most important product decisions — what to show on the card vs what to hide in the detail view.
Me
Should we show full OTE on every card or just a bracket? I'm worried that showing $264K–$314K everywhere makes it feel like a salary listing site, not a career platform.
Claude
Right instinct. Show brackets on cards ($200K–$320K), full comp only in the detail panel. Brackets signal the opportunity, full numbers reward the click. Every feature should either earn its place on the card or earn its place in the detail view — not both.
The general prompt template: We're deciding between [Option A] and [Option B] for [feature]. The concern with A is [specific problem]. What does each choice signal to the user, and which is more consistent with [brand goal]?
When to use this
Any time you're choosing between two product directions, frame it as a signal question. "What does this choice signal?" is more useful than "which is better?" — because better is subjective, but signal is about perception.
Part 2 — Building features that actually work
These prompts were used to build working functionality — not just mockups. The key difference is context. These include the real constraints upfront.
4
Setting up the Supabase connection Prompt Pattern
This is the prompt structure that got the database integration working cleanly.
Prompt used: I have a Supabase project. URL is [URL], anon key is [KEY]. The table is called Salesjobs (not "jobs"). RLS is enabled with public SELECT. Write a fetch() call that loads all active jobs from this table and prepends them to my existing JOBS array before render. Handle errors silently — if Supabase is down, fall back to the hardcoded jobs.
What made this work
Three things: giving the exact table name (the bug was "jobs" vs "Salesjobs"), specifying the error handling behaviour upfront ("silently"), and saying where the result goes ("prepend to JOBS before render"). Technical prompts need to specify the failure state, not just the happy path.
5
The "AI gate" decision Real Exchange
How the product went from "AI tools free on every card" to "AI tools gated behind waitlist."
Me
Browse jobs free, AI tools gated behind waitlist. Can we make the AI buttons on each card open a "coming soon, join waitlist" modal instead of the actual tools?
Claude
Yes, and that's the right product decision. The AI tools cost money per use (API calls), and exposing them before you have a payment wall means you pay for every anonymous visitor. Gate them behind waitlist now, open them when you launch Pro. The buttons stay on the cards as social proof of what the product does — they just redirect to waitlist instead of firing the API.
The lesson
Ask for a product decision, not just implementation. "Can we make X do Y" often gets a better answer if you also ask "is this the right decision?" The model knows enough context to flag when a decision has downstream consequences.
6
Mobile nav that actually works Prompt Pattern
Getting a proper mobile-first nav without fighting the layout.
Prompt used: Add a mobile bottom nav bar. Fixed to the bottom of the screen. Three items: Jobs (home icon), AI Tools (star), Waitlist (lock). Jobs links to the main job list. AI Tools and Waitlist open their respective modals. Match the white/navy aesthetic of the rest of the site. Hide it on desktop (display:none above 768px). The job list on mobile needs 80px of padding-bottom so the last card isn't hidden behind the nav bar.
The detail that matters
The padding-bottom fix is what most people miss. Every mobile bottom nav prompt should include "add X padding-bottom to the main content container." If you skip it, your last card disappears behind the nav and you spend 20 minutes debugging.
Part 3 — Writing that converts
The marketing copy, the origin story, the product positioning. These prompts produced the actual text on the live site.
7
The brand positioning prompt Real Exchange
This was the moment the product positioning clicked into something real.
Me
I moved to Canada from Nigeria at 17. First job out of university, the company went bankrupt. I Googled "which jobs make good money." Sales came up. I had no idea what I was doing. That's how it started.
Claude
That is gold. That is not just a personal brand story. That is a movement story. Every first-generation professional, every immigrant, every person who had to figure it out without a safety net will see themselves in that. The brand line is: Closed Won is for everyone who is building a sales career without a roadmap.
What made this land
Specific personal facts beat polished narratives every time. Don't say "I came from humble beginnings." Say "I Googled 'which jobs make good money' and that's how I ended up in sales." The AI doesn't know which facts are powerful — you do. Give it the raw material and let it find the pattern.
8
The launch post formula Prompt Pattern
The exact prompt that produced the Threads launch post.
Prompt used: Write a Threads launch post for tryclosedwon.com. Context: I moved to Canada from Nigeria at 17 with almost nothing. First job out of uni, company went bankrupt. Googled "which jobs make good money." Ended up in sales. Built a career to Head of New Business Sales at Google Canada. Built Closed Won while job searching. It's a sales job board with AI tools that tailor your resume, generate interview prep, and benchmark salary. Tone rules: - No therapy speak - No em dashes - No engagement bait hooks - UK English - Short sentences. One idea per line. - End with something honest, not a call to action. - The brand line somewhere in it: "For everyone who is figuring it out without a roadmap."
The tone rules section
The "tone rules" block at the bottom is what separates AI-generated copy from copy that sounds like you. Most people write the context and hope for the best. The rules section is where you eliminate the generic patterns — the "but here's the thing" hooks, the therapy-speak affirmations, the em dashes. Be specific about what you don't want.
9
Writing the hero headline Lesson
Why the headline on the site is "Find the role. Win the interview." and not something more clever.
Me
Give me 5 headline options for the hero. The product helps sales people find jobs and win interviews.
Claude
Options: 1. "Find sales roles. Win the interview." 2. "Your next sales role. Already prepared for." 3. "Search smarter. Interview sharper." 4. "The sales job board that preps you for the role." 5. "Built for closers. Designed to get you hired." The first one is the strongest. It's direct, it mirrors how people actually think about the job search (find it, then get it), and "Win" sets the tone for the brand without being cringe.
Ask for options, then ask which one
Always ask the model to both generate options AND recommend one. If you just ask for options, you get a list and have to make the call alone. If you ask for a recommendation with reasoning, you get a collaborator. The model's reasoning about why option 1 is strongest is often more useful than the headlines themselves.
Part 4 — Debugging and the stuff that breaks
These are the prompts from the moments things stopped working. The pattern: give the error, give the relevant code, give the expected behaviour.
10
The Supabase table name bug Real Exchange
The bug that took 20 minutes to find. The table name was "Salesjobs" not "jobs."
Me
Supabase fetch isn't returning anything. No error, just empty. The fetch call looks right but the jobs aren't loading.
Claude
Most likely causes in order: 1. Table name mismatch — is it "jobs" or "Salesjobs"? Supabase REST API is case-sensitive. 2. RLS policy not set — the policy needs to allow public SELECT. 3. The active filter is excluding all rows. Can you share the exact table name from your Supabase dashboard?
Me
The table is "Salesjobs" — the code says "jobs". That's it.
The debugging pattern that works
When something silently fails (no error, wrong result), the best prompt is: "Here's what I expected. Here's what happened. Here's the relevant code. Give me the most likely causes in order." The AI will triage before it fixes — which is faster than asking it to just "fix the bug."
11
Netlify 404 after deploy Real Exchange
The site deployed but the URL returned a 404. The fix was a file rename.
Me
Netlify deployed but I get a 404 at the URL. The file is called closedwon_v3.html.
Claude
Netlify looks for index.html as the root file. Rename closedwon_v3.html to index.html and push again. The deploy will auto-refresh within 30 seconds.
Keep the context in the prompt
The detail "the file is called closedwon_v3.html" is what made the diagnosis instant. Most debugging prompts fail because they describe the symptom but not the environment. Platform + symptom + file name = fast answer.
12
The universal ship-it prompt Lesson
The question that ended the session and put the site live.
Me
What do I actually need before I push this live? Be honest about what's blocking versus what I can fix later.
Claude
Blocking: rename the file to index.html, buy the domain, point DNS to Netlify. Not blocking: Stripe integration, email sequences, the full AI tools, 100 job listings. The product works now. Momentum matters more than perfection. Deploy it today, improve it publicly.
The most useful question you can ask
"What's blocking vs what can wait" is the most practical framing for any build decision. Most people ask "is this ready?" and get a vague answer. Asking "what specifically blocks launch?" forces a concrete list. The site went live the same day this question was asked.
